# 三句记忆卡纯 Vibe Coding 教程

这份文档只讲一件事：我是怎么通过一段真实对话，把一个三句记忆卡 demo 从想法一路聊出来、做出来、修出来的。

它不讲部署，不讲平台操作，不讲上线流程，只讲“和 AI 怎么对话、怎么一步步收敛需求、怎么把 demo 做成能用的样子”。

## 1. 先把想法说成一句话

一开始不要说“我要做一个系统”，而要说清楚你到底想解决什么。

这次最初的想法可以压成一句话：

> 我想做一个把每天发生的事压缩成三句话的记忆卡工具。

这句话很重要，因为它同时给出了三层边界：

- 这是一个“记录”工具，不是完整知识库
- 它强调“压缩”，所以每条内容都不能太长
- 它天然适合“回顾”，不是只写不看

这一步就是 vibe coding 的起点。不是先选技术，而是先把功能意图说到 AI 能立刻理解。

## 2. 再把需求压成最小可交付版本

真正好用的对话，不是一次性把所有需求都说完，而是先让 AI 先做一个最小版本。

这次最核心的交付边界就是：

- 单人使用
- 记录只有三句话
- 能增删改查
- 能搜索筛选
- 能随机回顾
- 能标记已回顾

我当时没有一上来要求复杂系统，而是先把 demo 定成“能记录、能看、能改、能删、能回顾”。

这样做的好处是：

1. AI 不会跑偏去做大而全产品。
2. 每次对话都能落到具体页面和按钮。
3. 你能很快判断 demo 有没有像样。

## 3. 先让页面长出来，再讨论技术细节

vibe coding 最有效的方式，不是把架构聊到完美，而是先让页面出现。

这次对话里，前端先做出来的东西包括：

- 顶部导航
- 新建卡片表单
- 卡片列表
- 卡片详情
- 搜索和筛选
- 随机回顾按钮

这一步的重点不是“代码是否最优”，而是“你有没有一个能看、能点、能感知产品形态的 demo”。

如果你用 AI 结对开发，建议你这样说：

- 先做一个简洁但完整的页面
- 每个按钮都先有位置
- 先有静态状态，再接交互
- 样式先够用，逻辑先跑通

这样 AI 会先帮你搭一个可见的壳，而不是一开始就陷进抽象架构。

## 4. 把“写卡片”当成核心交互

这个 demo 的灵魂不是列表，而是“写卡片”这件事。

所以在对话里，我们一直围绕以下几个问题推进：

- 用户写卡片的时候，最少要填什么
- 三句话怎么约束输入
- 标签和来源类型怎么放
- 重要程度怎么表达
- 保存之后如何立刻回到列表和详情

这些问题一旦说清，AI 就能把表单、状态和数据流慢慢搭起来。

这也是 vibe coding 的一个关键点：

> 不要让 AI 先写“框架”，而要先围绕用户的一个动作，把表单和状态聊明白。

## 5. 然后再接列表、筛选、详情这些周边能力

当写卡片这个核心动作成立以后，其他功能就好接了：

- 列表负责展示所有卡片
- 搜索负责缩小范围
- 筛选负责按来源类型和重要程度过滤
- 详情负责把一张卡完整展开
- 随机回顾负责增加使用感

这时候你和 AI 的对话就要从“页面长什么样”转成“状态怎么流动”。

比如你可以这样推动：

1. 列表里的卡片如何排序
2. 详情显示哪些字段
3. 搜索和筛选如何组合
4. 随机回顾应该从当前可见卡片里抽，还是从全部卡片里抽

这些讨论会把 demo 从“有个页面”推到“真的能用”。

## 6. 让 AI 帮你处理“增删改查”的一致性

三句记忆卡之所以像一个完整 demo，不是因为它有很多功能，而是因为增删改查很完整。

这部分最适合 vibe coding 的地方在于：

- 你不用每个接口都自己写死
- 你只要不断告诉 AI“这里的数据要同步”
- 然后让它帮你把前端状态和后端接口对齐

这次对话里，删除、编辑、标记回顾这些动作都被一点点补齐了。

你可以把这个过程理解成：

1. 先把“保存”做好
2. 再把“编辑”接上
3. 再把“删除”补上
4. 再把“回顾状态”补齐

每补一个动作，demo 就更接近真实产品一步。

## 7. 数据存储不是一开始就要定死

这份对话里有一个很典型的 vibe coding 过程：

- 先做起来
- 再发现存储不对
- 再换成更合适的方式
- 再继续让 demo 跑起来

也就是说，存储方案本身不是最早要想清楚的事情，先让产品跑起来更重要。

你可以把这类对话方式记成一句话：

> 先把 demo 做活，再把存储做稳。

## 8. 遇到错误时，不要先推倒重来，而是顺着错误找根因

对话里最有价值的一部分，其实是排错过程。

当删除失败时，我们不是直接重写整个项目，而是先看错误是什么，再往下追：

- 是前端没请求到接口
- 还是接口返回 500
- 还是写入逻辑碰到了只读文件

最后把问题定位到：代码还在尝试写本地文件。

这就是 vibe coding 里很重要的一种协作方式：

1. 先让 AI 帮你猜一个最可能的原因
2. 再用最小测试验证
3. 再顺着结果改最小的那一段

这样不会把项目越修越乱。

## 9. 对话里真正要保留的是“收敛过程”

如果你想把这份教程写得更像真实经历，那重点不该是最后用了什么，而是：

- 你先提了什么
- AI 先理解成什么
- 中间哪里改了方向
- 最后怎么收敛成 demo

这次三句记忆卡的聊天过程其实很典型：

1. 先是一个想法：做记忆卡
2. 然后变成页面：有表单、有列表、有详情
3. 再变成可用 demo：能增删改查、能回顾
4. 最后变成稳定版：旧代码删掉，逻辑统一

这条收敛路径就是 vibe coding 的核心。

## 10. 你可以怎样向 AI 继续推进一个类似项目

如果你以后再开一个类似项目，可以直接照这个对话节奏来：

1. 先说清楚产品一句话。
2. 再说最小版本必须有哪些按钮和页面。
3. 让 AI 先做静态页面。
4. 再让 AI 接表单和列表状态。
5. 再补保存、编辑、删除、回顾。
6. 最后让 AI 帮你删掉旧实现和多余代码。

你不用一次把所有要求讲完，只要每轮对话都比上一轮更具体，AI 就会越来越像一个跟你一起搭积木的同伴。

## 11. 这次 demo 是怎么被“聊”出来的

如果把整个过程压缩成一句人话，那就是：

我先告诉 AI 我想要一个“三句话记忆卡”，然后我不断补充“要能写、要能看、要能搜、要能删、要能回顾”，AI 就顺着这些对话，把它一点点拼成了一个真的可用 demo。

这就是这份教程真正想表达的东西：

> 不是先有完美方案，再去写代码；而是通过连续对话，把一个模糊想法一步步逼成一个能跑的产品。
